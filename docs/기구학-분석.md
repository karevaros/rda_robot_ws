# 기구학 분석 — 링크 파라미터 · 이동 제한 (4주차)

> 대상: **Python 컴포저**(`ros2 run rda_robot_assembler compose_urdf`)가 만든 통합 모델
> — Scout 2.0 + RB5-850e + RG2 + D405/D435i, 커밋된 `config/mounts.yaml` 기준(팔 mount z=0.11).
> 방법: **문서 전사 아님** — 실제 생성된 URDF 를 파싱해 추출하고, FK/정적토크로 검증한 값.
> 최종 업데이트: **2026-07-16** (tcp 96.7mm 오차 수정 반영 — 아래 2장 참조)

## 2. 링크 파라미터 (팔 체인)

zero-config 기준, 모든 관절 origin 의 rpy = 0 (순수 병진 오프셋).

| 관절 | type | parent → child | 축 | origin xyz (m) | 링크 오프셋 |
|------|------|----------------|-----|----------------|-------------|
| (mount) | fixed | `base_link` → `link0` | – | [0, 0, 0.10] | 0.100 m ※ `mounts.yaml` |
| `base` | revolute | `link0` → `link1` | +Z | [0, 0, 0.1692] | 0.1692 m |
| `shoulder` | revolute | `link1` → `link2` | +Y | [0, 0, 0] | 0 (동일원점) |
| `elbow` | revolute | `link2` → `link3` | +Y | [0, 0, 0.425] | **0.425 m** (상완) |
| `wrist1` | revolute | `link3` → `link4` | +Y | [0, 0, 0.392] | **0.392 m** (전완) |
| `wrist2` | revolute | `link4` → `link5` | +Z | [0, −0.1107, 0.1107] | 0.1566 m (손목 오프셋) |
| `wrist3` | revolute | `link5` → `link6` | +Y | [0, 0, 0] | 0 (동일원점) |
| (flange) | fixed | `link6` → `tcp` | – | **[0, −0.0967, 0]** | **0.0967 m** ← 2026-07-16 수정 |

- **구조:** 어깨/손목3 이 원점을 공유하는 전형적 6축 협동로봇 (spherical wrist).
- **reach 유도:** 0.425 + 0.392 + 0.1107 = **0.9277 m** — 제조사 제원 **927.7 mm 와 정확히 일치**.
  → 모델명의 "850" 은 reach 가 아님. (일부 유통사 페이지가 850 mm 로 표기하나 실측·제원 모두 927.7 mm.)
  → **이 유도는 `link6→tcp` 를 쓰지 않는다** — 그래서 아래 tcp 오차의 영향을 받지 않았고, 지금도 유효하다.

### ⚠ tcp 가 96.7 mm 틀려 있었다 (2주차~2026-07-16)

이 표의 flange 행은 2026-07-15 판까지 `[0, 0, 0]` 이었다. **버그였다.**
벤더의 `rb5_850e.urdf.xacro` 는 매크로를 `tcp_xyz="0 -0.0967 0"` 으로 부르는데,
당시 통합 URDF(`rda_robot.urdf.xacro`)가 **그 인자를 빠뜨려** 매크로 기본값 `'0 0 0'` 이
적용됐다 → tcp 가 `link6` 원점에 생겼다.

- **조립기는 정상이었다** — 벤더 standalone 을 그대로 로드하므로 tcp 오프셋이 살아 있었다.
  즉 **앱에서 맞춘 그리퍼 위치와 RViz/MoveIt 형상이 서로 달랐다.**
  (`mounts.yaml` 의 그리퍼 오프셋이 `[0,0,0]` 이라 보정 흔적도 없다 = 아무도 몰랐다.)
- **이 문서가 증상을 그대로 전사했다.** flange 행 `[0,0,0]` 은 "실측"이었지만, 측정 대상인
  URDF 자체가 틀렸다. 링크 파라미터 표만으로는 제원과 대조할 근거가 없어 넘어갔다.
- **해소:** Python 컴포저가 조립기와 **같은 벤더 소스**를 쓰면서 구조적으로 사라졌다.
  → SRDF/ACM 재생성 완료(`link4↔sensor1` 추가 / `sensor1↔sensor2` 제거 = tcp 이동의 귀결).
- **다른 팔도 제원과 대조해 검증했다** — RB10-1300e: 0.6127 + 0.5702 + 0.1172 = **1.3001 m**
  = 제원 1300 mm 일치. 같은 유도식이 성립하므로 컴포저 출력의 치수는 신뢰할 수 있다.

## 3. 이동 제한 (관절 제한)

| 관절 | lower | upper | 범위 | velocity | effort |
|------|-------|-------|------|----------|--------|
| `base`·`shoulder`·`elbow`·`wrist1`·`wrist2`·`wrist3` | −180° | +180° | 360° | 180°/s | 10 Nm |
| `rg2_finger_joint1` | 0° | 67.6° | 68° | 90°/s | 11 Nm |
| `rg2_finger_joint2` | mimic(`rg2_finger_joint1`) | | | | |

### ⚠ 검증 결과 — 값별로 신뢰도가 다름

벤더(`rbpodo_description/robots/rb5_850e/joint.yaml`)는 **6축 전부에 동일한 값**
(`lower −3.14 / upper 3.14 / effort 10 / velocity 3.14`)을 넣어두었다.
`rbpodo_moveit_config/config/joint_limits.yaml` 도 동일한 3.14 를 반복하고 가속도 제한은 꺼져 있다.
"전부 같은 값" 은 플레이스홀더 신호이므로 항목별로 따로 검증했다.

- ✅ **가동범위 ±3.14 rad 은 실제로 맞다.** 제원이 "360° rotation in all joints" 이므로 ±180°/축이 정확. (동일값이라고 다 가짜는 아님.)
- ✅ **링크 질량·관성도 타당.** URDF link1~6 질량 합 20.84 kg vs 제원 전체 22 kg (베이스 커버 등 제외 시 합리적).
- ❌ **effort 10 Nm 은 명백한 플레이스홀더.** URDF 자체 질량으로 정적 토크를 계산하면:

  | 기여 | 질량 | 모멘트암 | 토크 |
  |------|------|----------|------|
  | link2 | 9.63 kg | 0.335 m | 31.6 Nm |
  | link3 | 3.92 kg | 0.643 m | 24.7 Nm |
  | link4 | 1.45 kg | 0.949 m | 13.5 Nm |
  | link5 | 1.45 kg | 1.060 m | 15.1 Nm |
  | link6 | 0.24 kg | 1.118 m | 2.7 Nm |
  | 페이로드(정격 5 kg) | 5.00 kg | 1.135 m | **55.7 Nm** |
  | **shoulder 필요 정적토크(팔 수평 신장)** | | | **≈ 143 Nm** |

  → URDF 값(10 Nm)의 **약 14배**. **정격 페이로드 단독(55.7 Nm)만으로도 10 Nm 를 넘는다** = 자기 팔조차 들 수 없는 값.

  > **2026-07-16 정정 (133 → 143 Nm).** 두 가지가 겹쳐 있었다.
  > ① tcp 96.7mm 오차(2장) ② `torque_check.py` 가 **링크와 페이로드에 서로 다른 식**을 썼다 —
  > 링크는 `|Δz|+|y|+|x|`(측방 오프셋 포함)인데 페이로드만 `|Δz|` 로 z 차이만 봤다.
  > tcp 오프셋이 **측방**이라 이 불일치 때문에 페이로드 토크만 영향을 안 받고 있었다.
  > 식을 링크와 일치시켰다(모멘트암 0.928 → 1.135 m). **결론은 그대로** — 10 Nm 는 플레이스홀더.
  >
  > 참고 — 현재 `mounts.yaml` 이 고른 **RB10-1300e**(정격 10 kg)로 같은 계산: **≈337 Nm**(33.7배).
- ⚠ **velocity 180°/s 은 미검증.** 6축 동일값이라 의심되나, 공개 제원표(유통사·카탈로그)에 축별 속도가 없어 확증 불가. 실제 제어기 연동(6주차) 전 **제조사 데이터시트로 확인 필요.**
- ❌ **가속도 제한이 아예 없다** (`has_acceleration_limits: false`).

### 실무 영향 (어디까지 문제인가)

| 용도 | effort 10 Nm | 가속도 누락 |
|------|--------------|-------------|
| 기하학적 경로계획(OMPL, 4~5주차) | 영향 없음 (토크 미사용) | 영향 없음 |
| **궤적 시간 파라미터화(TOTG)** | 영향 없음 | ❌ **직접 영향** — 가속도 제한 없이는 시간최적화 불가 |
| 동역학 기반 제어·토크 검증(6주차) | ❌ 사용 불가 | ❌ 사용 불가 |
| Gazebo 물리 시뮬(7주차) | ❌ 팔이 중력에 무너짐 | ⚠ |

→ **4~5주차(장애물·경로계획)에는 지장 없음.** 단 **가속도 제한은 5주차 궤적 생성 전에 반드시 채워야 한다.**
→ effort 는 6주차(통합제어)·7주차(가상환경) 전에 데이터시트 값으로 교체 필요.

## 4. 작업 영역 (workspace)

`base_link` 기준, 팔 6축 균등 몬테카를로 샘플 200,000 회 FK.

| 항목 | 값 | 2026-07-15 판 |
|------|-----|--------------|
| 팔 베이스(`link0`) 위치 | [0, 0, **0.11**] ※ `mounts.yaml` 값 | [0, 0, 0.10] |
| tcp 반경 (link0 기준) | min 0.015 m / **max 1.138 m** | 0.111 / 1.102 |
| X 범위 | −0.962 ~ +0.959 m | −0.932 ~ +0.930 |
| Y 범위 | −0.959 ~ +0.964 m | −0.929 ~ +0.931 |
| Z 범위 | −0.684 ~ **+1.240 m** | −0.658 ~ +1.196 |
| 홈포즈(전관절 0) tcp | [**0.207**, 0, 1.207] | [0.111, 0, 1.197] |

> 두 판의 차이는 **① tcp 96.7mm 수정**(2장)과 **② 팔 mount z 가 0.10 → 0.11 로 바뀐 것**이
> 겹친 결과다. 홈포즈 tcp 의 x 가 0.111 → 0.207 로 뛴 게 tcp 오프셋이 드러난 자리다
> (오프셋은 손목 기준 측방이라, 홈포즈에서 x 로 나타난다).
> tcp 최대반경 1.102 → 1.138 은 **mount 와 무관**하다(link0 기준 상대값) = 순수한 tcp 수정 효과.

### ⚠ 이동 제한 영역이 필요한 근거
- **샘플의 26.4% 가 z < 0** — 즉 관절 제한(±180°)만으로는 **tcp 가 바닥(그리고 모바일 플랫폼 상판)을 파고드는 자세를 막지 못한다.** (RB10-1300e 로 바꾸면 **30.4%** 로 더 나빠진다.)
- 자충돌 검사(`self_collision_monitor`)는 **링크끼리의 충돌**만 본다. **바닥면은 URDF 에 링크가 없어 검출 대상이 아니다.**
- → 4주차 "이동 제한 영역" 은 **바닥 평면 + 플랫폼 상판**을 장애물/제한영역으로 명시해야 실효가 있다. (`obstacles.yaml` 에 ground plane 포함)

## 5. 재현 방법

```bash
DESC=~/robot_ws/src/rda_robot_description
ros2 run rda_robot_assembler compose_urdf --mounts $DESC/config/mounts.yaml -o /tmp/rda_robot.urdf
python3 src/docs/scripts/kinematics.py   /tmp/rda_robot.urdf     # 링크 파라미터·제한·작업영역
python3 src/docs/scripts/torque_check.py /tmp/rda_robot.urdf     # effort 타당성 정적토크 검증
```

**다른 팔로 바꿔 재현하려면** 링크 접두사를 넘긴다. 내장 RB5 는 무접두(링크명 계약)지만
폴더 드롭 모델은 슬롯 접두사가 붙는다(`compose_urdf` 의 prefix 규칙):

```bash
# 예: arm__rb10_1300e (정격 10 kg)
python3 src/docs/scripts/kinematics.py   /tmp/rda_robot.urdf arm_tcp arm_link0
python3 src/docs/scripts/torque_check.py /tmp/rda_robot.urdf arm_ 10
```

> ⚠ 4장 작업영역은 `config/mounts.yaml` 의 **팔 mount z 에 의존한다**(이 문서 = 커밋된 값 0.11 m).
> mount 나 팔 모델을 바꾸면 **재산출 필요** — 2026-07-15 판이 0.10 기준이었는데 mount 가 0.11 로
> 바뀐 뒤 이 문서가 갱신되지 않아 값이 어긋나 있었다.

## 6. ⚠ 벤더 SRDF 를 그대로 쓰면 안 되는 이유 (4주차 MoveIt 셋업)

`rbpodo_moveit_config/config/rbpodo.srdf` 는 팔 단독 SRDF 로 `disable_collisions` 12쌍을 정의한다.
그중 **`reason="Never"`(=어떤 자세에서도 충돌 안 함) 6쌍을 우리 통합 URDF 로 재검증**한 결과
**4쌍이 실제로 충돌**했다 (`docs/scripts/gen_srdf.py`, 무작위 자세 샘플링).

| 쌍 | 벤더 주장 | 실측(샘플 중 충돌 비율) | 판정 |
|----|-----------|------------------------|------|
| `link0`↔`link2` | Never | 0 % | ✅ 일치 |
| `link4`↔`link6` | Never | 0 % | ✅ 일치 |
| **`link0`↔`link3`** | Never | **8.3 %** | ❌ **틀림** |
| **`link1`↔`link3`** | Never | **8.3 %** | ❌ **틀림** |
| **`link3`↔`link5`** | Never | 충돌함 | ❌ **틀림** |
| **`link3`↔`link6`** | Never | 충돌함 | ❌ **틀림** |

통계가 아니라 **구체적 관절값으로 재현 확인**:

| 쌍 | 실제 충돌하는 자세 (deg) |
|----|--------------------------|
| `link1`↔`link3` | base −42.7, shoulder 1.4, **elbow −174.0**, wrist1 −2.3, wrist2 169.8, wrist3 −77.2 |
| `link0`↔`link3` | base 19.3, **shoulder 153.3**, elbow −179.3, wrist1 −121.6, wrist2 79.3, wrist3 −38.0 |
| `link3`↔`link5` | base −62.5, shoulder 160.1, elbow 167.8, wrist1 177.3, wrist2 −164.7, wrist3 117.5 |
| `link3`↔`link6` | base 89.4, shoulder −20.6, elbow −104.7, wrist1 145.8, wrist2 −173.9, wrist3 −70.7 |

`elbow −174°` = 팔을 자기 자신 쪽으로 접은 자세 — 명백한 실제 충돌이다.

→ **벤더 SRDF 를 복사해 쓰면 MoveIt 이 이 4쌍의 자충돌을 무시하고 경로를 낸다.**
→ 그래서 `docs/scripts/gen_srdf.py` 로 **우리 통합 URDF 기준 ACM 을 직접 산출**한다.
  GUI(Setup Assistant) 대신 스크립트인 이유: 통합 URDF 가 `mounts.yaml`(조립기)로 바뀌므로
  모델 교체 시마다 재생성해야 하고, GUI 1회 산출물은 곧 낡는다.

**주의 — ACM 은 반드시 collision mesh 로 산출할 것.** MoveIt 은 URDF 의 collision geometry 로
검사한다. visual mesh 로 ACM 을 만들면(더 얇음) 실제로는 충돌하는 쌍을 `Never` 로 꺼버린다.
실측 차이: visual 기준 무시 66쌍 → collision 기준 63쌍 (**3쌍이 뒤집힘**).
(yourdfpy 는 `build_collision_scene_graph=True` 여야 `collision_scene` 이 생긴다. 기본은 `None`.)
